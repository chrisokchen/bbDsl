# Bridge Bidding System DSL

> 叫牌制度的設計

這太適合你了——把你在規格語言設計上的專長直接遷移到橋牌領域。我們來好好構思這個 **Bridge Bidding DSL (BBDSL)**。

## 先釐清幾個設計問題

在動手之前，想先跟你對齊幾個關鍵決策：

**目標用戶是誰？**
- 純粹你個人和搭檔使用？
- 還是希望做成一個通用工具，讓橋牌社群也能用來描述任意制度？
- 或者也考慮教學用途（例如教孫女時，用視覺化的方式呈現叫牌邏輯）？

這會影響語言的複雜度與抽象層級。

**你偏好的技術路線？**
- 基於 YAML/JSON 的宣告式描述（和你的 haPDL 經驗一致）
- 自訂語法的 DSL（更簡潔但需要 parser）
- 或者先用 Markdown + 結構化標記的混合方式快速原型

**你希望 DSL 能驅動什麼？**
比如：
- 自動產生約定卡（Convention Card）文件
- 一致性驗證（偵測遺漏或衝突的叫牌序列）
- 模擬練習題產生器
- 與 AI 對練時作為制度知識庫

這些下游應用會決定 DSL 需要捕捉的語義粒度。

## 設計思路

先從橋牌叫牌的**領域模型**出發，識別核心概念：

**叫牌制度的本質是什麼？**
一棵巨大的決策樹——根據「前序叫牌 + 手中牌的條件」映射到「下一個叫品」。但它不是單純的樹，因為：

1. **上下文敏感**：同一個叫品在不同序列中意義不同（1♣-1♦-1♥ 中的 1♥ vs 1♦-1♥ 中的 1♥）
2. **條件是多維的**：HCP、牌型、控制、特定花色長度等
3. **有競叫干擾**：對手的叫牌會改變整個語境
4. **有約定層級**：從基礎制度到特約（convention），是可組合的模組

這和你在 WA-RAPTor 處理的「多目標規格轉換」問題結構很像。

讓我來草擬第一版 schema：那不是我需要的 skill。讓我直接開始設計 BBDSL schema。現在建立對應的 JSON Schema，讓 YAML 可以被機器驗證：再建一個精簡的範例檔，讓人一眼看懂 BBDSL 怎麼用：現在把所有檔案複製到 outputs：這是 BBDSL 的第一版草案，包含三個檔案：

1. [**BBDSL-SCHEMA.md**](BBDSL-SCHEMA.md) — 完整的規格說明文件，以精準制為範例，涵蓋概念模型、schema 結構、設計原則和後續 roadmap
2. [**bbdsl-schema.json**](bbdsl-schema.json) — JSON Schema 定義，讓 YAML 檔可被機器驗證
3. [**sayc.bbdsl.yaml**](sayc.bbdsl.yaml) — 用 SAYC（自然制）做的精簡範例，展示實際寫法

幾個關鍵設計決策想跟你確認：

**叫牌序列的表達方式**：目前用巢狀結構（responses 裡面再套 continuations），好處是結構清楚，壞處是深層叫牌序列會縮排很深。另一個選項是用扁平化的路徑表示法，例如 `path: "1C-1D-1H"` 來定義任意深度的序列。你偏好哪種？

**牌型描述的精確度**：目前 `shapes: ["4-3-3-3"]` 中的四個數字是「由大到小排列的花色長度」，不指定具體花色。需要指定具體花色時用 `clubs/diamonds/hearts/spades` 的個別約束。這樣的分層夠用嗎？

**Convention 的觸發條件**：目前用簡單的 `trigger.after` 陣列，但實際橋牌中觸發條件可能很複雜（例如「對手在搭檔開叫 1H 後爭叫 2D，我方叫 2NT」）。這部分需要設計更完整的序列匹配語法嗎？
