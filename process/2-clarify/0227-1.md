● 讀完了整份實作計畫書。以下是我認為需要澄清的問題：

  ---
  1. 授權不一致

  計畫書裡 LICENSE 寫 CC-BY-SA-4.0，但 repo 實際的 LICENSE 檔案是 CC0 1.0。程式碼授權要用哪一個？這會影響社群貢獻和
  Convention Registry 的法律基礎。

  2. 套件管理工具的選擇

  技術選型寫「uv / pyproject.toml」，但 uv 和 poetry 的 lockfile 格式和工作流差異很大。你要用哪一個？這決定了 CI/CD
  設定和貢獻者的 onboarding 流程。

  3. Phase 1 驗證規則數量矛盾

  - 總覽圖寫「驗證器 (8 規則)」
  - Sprint 1.2 任務清單列了 val-002, 004, 005, 006, 007, 008, 011, 012 = 8 條
  - 但 Phase 1 完成標準又說「valid/ 全部 PASS，invalid/ 全部偵測到預期的錯誤」

  val-001 (hcp-coverage) 和 val-003 (response-complete) 被推到 Phase 2，那 Phase 1
  的精準制範例驗證時碰到這兩條怎麼處理？直接跳過、還是先產出 warning？

  4. _unresolved 和 _bml_original 的定位

  Pydantic 模型裡 _unresolved 用底線開頭（Python 的 private convention），但 Pydantic v2 對 _ 前綴的 field
  有特殊行為（不會在 serialization 時輸出）。這些欄位是：
  - 只在匯入流程中暫存，最終不寫入 YAML？
  - 還是會持久存入 .bbdsl.yaml 給使用者手動補完？

  如果是後者，欄位名稱不應該用底線前綴。

  5. OpponentPattern 的 matches() 方法

  模型裡放了 def matches(self, opponent_action: dict) -> bool 但標註 Phase 2 實作。Pydantic BaseModel
  裡放業務邏輯方法是有意為之的設計，還是打算後續把匹配邏輯移到 core/ 的獨立 service？

  6. BML 匯入的範圍定義

  BML 本身沒有正式規格（只有 convention），不同作者的 BML 寫法差異很大。計畫書目標「HCP/花色長度正確提取率 > 80%」——這個
   80% 是基於哪些 BML 樣本測量的？有沒有預定的測試語料庫？

  7. Phase 5 技術棧跳躍

  Phase 1-4 全部是 Python CLI 工具，Phase 5 突然出現：
  - FastAPI + PostgreSQL（後端）
  - React + Monaco Editor（前端）
  - WebSocket 即時通訊
  - OAuth 使用者系統

  這等於是另起一個全端 web 專案。有考慮過 Phase 5 獨立成另一個 repo 嗎？或者你打算 monorepo 處理？

  8. 單人開發的優先級取捨

  風險表裡已經指出「單人開發瓶頸」機率高。32 週的計畫涵蓋 CLI + 匯入匯出 + 視覺化 + AI 模擬 +
  社群平台，範圍非常廣。如果時間不夠，Phase 3/4/5 之間的優先順序是什麼？例如：AI 整合 vs
  視覺化，哪個對你的目標使用者更重要？

  ---
  這些不影響 Phase 1 開工，但越早定下來，後面的架構決策越穩。你想先回答哪些？
