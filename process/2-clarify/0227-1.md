  ==> 請明列這兩種的不同，會怎麼影響社群貢獻和 Convention Registry 的法律基礎?

  這是一個非常關鍵且常見的開源專案決策！授權條款（License）不僅是法律文件，更是專案對社群傳達的「價值觀與遊戲規則」。

首先要釐清一個基礎概念：Creative Commons (CC) 系列授權條款，官方強烈不建議用於「軟體程式碼」（因為 CC 沒有處理專利權以及源碼編譯等軟體特有問題）。因此，我們通常會將專案分為「程式碼 (Code)」與「資料/文本 (Data/Content)」來討論。

以下為您明列 CC0 1.0 與 CC-BY-SA-4.0 的核心差異，以及它們如何影響您的橋牌 DSL 專案、社群貢獻與 Convention Registry：

1. 核心差異明細

| 特性 | CC0 1.0 (對應程式碼的 MIT/Unlicense) | CC-BY-SA 4.0 (對應程式碼的 GPL 系) |
| --- | --- | --- |
| 定義 | 零版權 (公眾領域)：放棄所有著作權。 | 姓名標示-相同方式分享：保留著作權但開放使用。 |
| 使用限制 | 無限制。任何人皆可複製、修改、散布、商業使用。 | 需標示原作者 (BY)；若修改或衍生，必須以相同授權釋出 (SA)。 |
| 商用友善度 | 🌟🌟🌟🌟🌟 極高 | ⭐ 低 (具「傳染性」，商業公司通常避之唯恐不及) |
| 對衍生作要求 | 不要求開源修改後的心血。 | 強制開源 (Copyleft 特性)。 |

2. 對「社群貢獻」的影響
這涉及到您希望建立什麼樣的開源生態：

* 若選擇 CC0 (或 MIT)：
  - 優點：門檻極低，推廣阻力最小。學術界、商業橋牌平台（如 BBO 或是其他橋牌軟體公司）可以直接把您的 Parser 整合進他們的核心系統中，不用擔心法律問題。
  - 缺點 (社群反噬)：別人可以 fork 您的程式碼或註冊表，加上超棒的新功能然後閉源收費。他們沒有義務將修改回饋給您的主線 (Upstream)。
* 若選擇 CC-BY-SA (或 GPL/AGPL)：
  - 優點：保障「公共財」。任何開發者、廠商只要拿您的專案去修改或擴充，就**「必須」開源並貢獻回社群**。這能確保您的專案不會被微軟或某個大型橋牌平台給「白嫖」後私有化。
  - 缺點：許多商業專案的法務會禁止工程師引入 SA (ShareAlike) 或 GPL 授權的工具，這可能會嚴重限制您的解析器在業界被採用的廣度。

3. 對「Convention Registry」法律基礎的影響
在您的 Phase 5，會建立一個橋牌約定 (Bidding Conventions) 的 Registry。這是文字與資料庫 (Data)，非常適合使用 CC 授權。這兩種授權會造就完全不同的發展走向：

* 如果 Registry 採 CC0：
  - 您在打造橋牌界的 Wikipedia 早期。任何人都可以無痛把這些約定資料庫抓下來，放進自己的付費 App 裡當成賣點。這有利於把您的 bbDsl 格式推廣成世界絕對標準，但您和社群可能分不到商業利益。
* 如果 Registry 採 CC-BY-SA：
  - 您在打造橋牌界的 Wikipedia (維基百科正是使用 CC-BY-SA 授權)。如果某個社群大神貢獻了極其複雜的「精準制 (Precision) 變體」，任何平台想要內建這個變體，也必須把他們衍生的任何規則開源出來。這能築起極高的「資料護城河」，確保 Registry 的知識量永遠獨步全球且屬於全人類。

1. pyproject.toml 就表示用 Poetry 嗎？
不是的。

過去 Python 專案通常習慣用 setup.py 或 requirements.txt 來管理套件。但後來 Python 官方推動了標準化（PEP 518 和 PEP 621），將 pyproject.toml 制定為所有 Python 專案通用的設定檔標準。

現在，幾乎所有現代的 Python 工具共用這個檔案：

* 建置與依賴管理工具：Poetry, uv, Hatch, PDM, Flit 都讀取這個檔案（但各自會在裡面建立自己的專屬區塊 [tool.poetry] 或 [tool.uv]）。
* 程式碼品質工具：Ruff, Black, mypy, pytest 等也把設定寫在這個檔案裡的 [tool.ruff], [tool.pytest] 區塊。

所以看到 pyproject.toml，只代表這個專案是「現代化的 Python 專案」，並不直接等同於使用 Poetry。在您的計畫書中寫「uv / pyproject.toml」，意思是「使用 pyproject.toml 作為標準設定檔，並使用 uv 來作為核心的套件與環境管理工具」。

2. 這怎麼決定了 CI/CD 設定和貢獻者的 onboarding（報到/上手）流程？
當您決定要用 uv 或是 poetry 時，會產生截然不同的檔案結構與指令，這會直接影響別人怎麼參與你的專案，以及自動化腳本怎麼寫：

A. 對「貢獻者 Onboarding (新手上手流程)」的影響
貢獻者 clone 您的 repo 後，第一件事就是把環境架起來。

* 如果用 Poetry：
  * 貢獻者需要先安裝 Python： 他們得自己想辦法在電腦上裝好 Python (例如用 pyenv)。
  * 上手指令：貢獻者需要先按官方文件安裝 Poetry，然後執行 poetry install。Poetry 會自動建立虛擬環境 (.venv) 並安裝 poetry.lock 裡的套件。
  * 體驗：Poetry 生態系很成熟，指令直覺（poetry add、poetry run），但解析龐大依賴關係時有時會有點慢。
* 如果用 uv (推薦，這也是您原本計畫書的方向)：
  * 貢獻者不需要先裝 Python： uv 本身連 Python 版本都能幫你抓！貢獻者只要裝了 uv（一個用 Rust 寫的極速工具），執行指令，uv 會自動幫你下載正確版本的 Python。
  * 上手指令：貢獻者執行 uv sync。
  * 體驗：速度快到不講武德（通常比 Poetry 快 10-100 倍）。對於新手來說，不用處理「我的 Python 版本不對」的問題，是很棒的體驗。不過 uv 的專案管理功能（uv.lock 等）是 2024 下半年才剛推出的新功能，社群文章相對 Poetry 少一點點。

B. 對「CI/CD 設定 (如 Github Actions)」的影響
在做自動化測試或打包佈署時，CI 系統需要知道用什麼工具來還原環境。

* 如果用 Poetry：
  * GitHub Actions 裡需要先 actions/setup-python 安裝 Python，再安裝 Poetry 工具，利用 poetry.lock 來快取依賴。
  * 指令：poetry run pytest (跑測試)。
* 如果用 uv：
  * GitHub Actions 直接使用 astral-sh/setup-uv 這個 action。因為 uv 速度極快，甚至連依賴快取 (caching) 的設定有時都可以省了，因為它重新安裝可能只需要不到一秒鐘。
  * 指令：uv run pytest。
  * 您在 repo 中會提交一個叫做 uv.lock 的檔案（就如同 Node.js 的 package-lock.json），來確保 CI 上的套件版本和您本機一模一樣。

總結建議
以您原本計畫書的方向，建議 明確宣示採用 uv（搭配 pyproject.toml 與 uv.lock）。

為什麼？ 因為 uv 目前在 Python 社群是當紅炸子雞（由開發 Ruff 的同一個團隊 Astral 製作）。它統一了 pip, venv, pyenv, pip-tools 甚至部分 poetry 的功能。這能讓未來的貢獻者（或您自己在其他機器上）只要一行 uv sync，就能在幾秒內瞬間把一模一樣的開發環境架好，開發體驗非常絲滑！

  ==> 直接跳過、還是先產出 warning 分別會有什麼影響?

這是一個非常務實的架構設計考量。

在驗證器（Validator）的設計中，val-001 (hcp-coverage 點力覆蓋率) 和 val-003 (response-complete 應叫完備性) 屬於 「語意與邏輯完備性 (Semantic/Logical Completeness)」 的深度檢查。它們比單純檢查「格式有沒有寫錯」要困難得多。

如果在 Phase 1 碰到這兩條規則，以下是兩種處理方式的具體影響與優缺點：

### 選項 A：直接跳過 (Silently Skip / Ignore)
也就是在程式碼中完全不註冊、不執行這兩條規則，當作它們還不存在。

* 對開發的影響 (優點)：Phase 1 的開發最乾淨俐落。您可以百分之百專注在基礎語法（Syntax）、不合法的叫品、重複條件等最基本的解析工作，不用去碰樹狀結構遍歷或點力區間交集的複雜演算法。
* 對範例與測試的影響 (缺點 - 虛假的安全感)：您在 Phase 1 寫的「精準制範例檔 (valid/precision.yaml)」，會順利得到 ✅ All validations passed.。但實際上，這個檔案可能存在嚴重的邏輯漏洞（例如漏規定了 15-17 點的開叫，或漏了對手插叫後的處理）。當您推進到 Phase 2 且實作了這兩條規則時，原本 PASS 的範例檔會突然吐出一大堆 Error，導致需要回頭大幅修改 Phase 1 寫好的 YAML 檔。

### 選項 B：先產出 Warning (Stub & Warn)
Phase 1 就定義好這兩條規則的 ID，但不進行真正的運算檢查，僅僅是拋出一個寫死的 Warning（警告），不把這視為會導致中斷的 Error。

* 對架構的影響 (優點 - 建立分級機制)：編譯器或驗證器早晚都需要區分 Error (致命語法錯誤，無法繼續) 與 Warning (不好的習慣或潛在的漏洞，但仍可編譯)。這樣做可以在 Phase 1 就強迫您把「錯誤分級機制」建立起來。
* 對使用者/測試的影響 (優點 - 管理期望)：跑 Phase 1 的驗證時，輸出會像是這樣：

```text
> bbDsl validate precision.yaml
[OK] val-002: Syntax is correct.
[OK] val-005: No duplicate condition branches.
[WARN] val-001 (WIP): HCP coverage check is skipped in Phase 1.
[WARN] val-003 (WIP): Response completeness check is skipped in Phase 1.
Result: ✅ Passed with 2 Warnings.
```

這樣能時刻提醒開發者「這份檔案格式對了，但邏輯不保證無死角」。
* 對開發的影響 (缺點)：會讓終端機輸出增加一些「雜訊」，而且您需要在 Phase 1 就設計好一套能收集、彙整並漂亮印出 Warning 的輸出格式。

4. _unresolved 和 _bml_original 的定位
`
  Pydantic 模型裡 _unresolved 用底線開頭（Python 的 private convention），但 Pydantic v2 對 _ 前綴的 field
  有特殊行為（不會在 serialization 時輸出）。這些欄位是：
  - 只在匯入流程中暫存，最終不寫入 YAML？
  - 還是會持久存入 .bbdsl.yaml 給使用者手動補完？

  如果是後者，欄位名稱不應該用底線前綴。
  ==> 請分析二者的利弊得失，給我建議

  這是一個關於「轉檔器/編譯器體驗 (Developer Experience)」的經典工程問題。針對處理不規則來源資料（如 BML），這兩個方向代表了截然不同的設計哲學。

以下為您分析這兩條路線的利弊，並給出強烈建議。

### 路線一：作為暫存狀態，不寫入 YAML（也就是保持 _ 前綴）
當 BML 匯入器遇到看不懂的句子時，在記憶體中標記為 _unresolved，但在最終輸出 .bbdsl.yaml 檔案時將其丟棄（或只在終端機印出警告）。

* 優點 (Pros)：
    * Schema 絕對純淨：您的 bbDsl 規範非常乾淨，不會染上「轉檔過程的過渡產物」。
    * 驗證邏輯簡單：Validator 不用去處理「欄位存在但內容是未知」的邊界情況。YAML 只要產得出來，結構就一定是 100% 正確的。
* 缺點 (Cons)：
    * 可怕的「靜默資料遺失 (Silent Data Loss)」：使用者匯入了一份 500 行的 BML 檔，結果輸出的 YAML 只有 450 行。那 50 行包含重要打牌約定的資料憑空消失了，使用者很難查覺到底是哪幾條沒被解析出來。

### 路線二：作為持久化欄位，寫入 YAML（必須移除 _ 前綴，例如改成 unresolved 和 bml_original）
當解析器遇到看不懂的 BML 句子時，依然照樣產生一個 YAML 節點，但裡面不放結構化資料，而是把原始字串塞進 bml_original 並標記為 unresolved: true。

* 優點 (Pros)：
    * 零資料遺失 (Zero Data Loss)：這對轉檔工具是最高指導原則。使用者絕對不想重新翻找原始 BML 檔。
    * 極佳的遷移體驗：使用者可以直接打開匯出後的 .bbdsl.yaml，用編輯器全域搜尋 unresolved:，然後看著旁邊完整的 bml_original 提示，逐一用手動方式補上正確的 DSL 語法。這比起看着終端機的一閃而過的錯誤訊息來得實用太多了。
* 缺點 (Cons)：
    * 污染了標準 Schema：您的正式 Pydantic 模型裡，必須容忍 unresolved 這種「不完美」的欄位存在。
    * Validator 必須跟著改：驗證器在檢查檔案時，必須特別寫一段邏輯：遇到 unresolved: true 的節點時，要拋出特定的編譯錯誤（例如：Error: Found unresolved BML rule at line 45. Please fix it manually.）。

###💡 最終建議與架構解法
強烈建議採用「路線二」（持久化寫入 YAML）。

橋牌的 BML 缺乏嚴格規範（有時甚至夾雜人類自然語言註解），自動解析器絕對不可能達到 100% 成功率。如果採取路線一，這個匯入工具的實用性會大打折扣。

但在 Pydantic 架構上，為了不讓「過渡期欄位」弄髒您乾淨的 DSL 標準，建議採用以下**「繼承/多型分層」**的解法：

具體實作建議
不要在標準核心的 CallNode 裡面放這兩個欄位，這會混淆標準格式。您可以把「未解析節點」做成一種特定的「錯誤類型節點」：

1. 修改 Pydantic 結構 (移除底線，變成合法 Node)

```python
from pydantic import BaseModel, Field
from typing import Union, List
# 1. 您原本乾淨的標準節點
class CallNode(BaseModel):
    call: str
    condition: str | None = None
    meaning: str | None = None
    next: List["Node"] = []
# 2. 新增一種專門用來接住匯入失敗的「未解析節點」
class UnresolvedNode(BaseModel):
    is_unresolved: bool = Field(default=True, const=True) # 標記
    bml_original: str  # 留下證據
    reason: str        # 為什麼解析失敗（例：不明的特殊符號）
# 3. 樹狀結構允許這兩種節點混居
Node = Union[CallNode, UnresolvedNode]
```

2. 這個設計帶來的完美工作流：

* 匯入時 (BML -> YAML)：解析器盡力而為。懂的變成 CallNode，不懂的退化成 UnresolvedNode，一起寫入 YAML。
* 使用者手動修復 (Manual Fix)：使用者打開 YAML，看到：

```yaml
- call: 1S
  next:
    - is_unresolved: true
```

  5. OpponentPattern 的 matches() 方法

  模型裡放了 def matches(self, opponent_action: dict) -> bool 但標註 Phase 2 實作。Pydantic BaseModel
  裡放業務邏輯方法是有意為之的設計，還是打算後續把匹配邏輯移到 core/ 的獨立 service？
  ==> 這題的建議做法?

這是一個非常深入核心軟體工程設計（也就是 DDD 架構 - 領域驅動設計）的問題。

我們來看看您提的問題：

OpponentPattern 模型裡面放了 def matches(self, opponent_action: dict) -> bool。 Pydantic BaseModel 裡面放業務邏輯方法是有意為之，還是打算後續移到獨立 service？

在系統架構設計上，這兩種做法分別代表了 「Rich Domain Model (充血模型)」 與 「Anemic Domain Model (貧血模型) + Service」 的路線。

以下是針對這個問題的分析與強烈建議：

###選項 A：留在 Pydantic 內（Rich Domain Model 充血模型）
這是您計畫書目前寫的狀態。資料模型 (BaseModel) 不只儲存資料，還擁有操作這些資料的行為（方法）。

* 優點 (Pros)：
    * 高內聚 (High Cohesion)：關於「什麼是對手模式」以及「如何判斷一個行為符合該模式」，這些邏輯被緊緊綁在同一個類別裡。不管在哪裡呼叫，都不用再引入其他的輔助模組。
    * 直覺的物件導向：呼叫起來像自然語言 pattern.matches(action)。
* 缺點 (Cons)：
    * 違反單一職責 (SRP)：Pydantic 的主要責任是「資料驗證與解析」。如果在裡面塞滿複雜的遞迴比對邏輯、正規表達式與演算法，這個類別會變得很雜亂。
    * 耦合過重：如果要比對的過程需要存取其他的依賴項（例如：需要查閱 SUIT_META 或 Definitions 來知道 majors 代表什麼），放在模型內部就很難注入這些外部設定。

### 選項 B：移到獨立的 Service（Anemic Domain Model 貧血模型）
資料模型（Pydantic）只負責定義資料的形狀與驗證語法，而所有的業務邏輯（如 matches）都被抽離到 core/ 下的獨立函數或類別（例如 PatternMatcher）。

* 優點 (Pros)：
    * 職責分離 (Separation of Concerns)：models/context.py 只管資料乾不乾淨，core/matcher.py 只管演算法怎麼跑。這對測試非常友善。
    * 支援複雜上下文：如果比對時需要知道「現在是打什麼身價」或是「前面的人叫了什麼」，獨立的 Service 可以很容易地把整個 Context 或是 Definitions 物件當作參數傳進去。
* 缺點 (Cons)：
    * 呼叫稍微囉嗦：語法變成 PatternMatcher.matches(pattern, action, context)。

### 💡 最終建議：採用 選項 B (移到獨立 Service)
在您的橋牌 DSL 這個特定情境中，把邏輯從 Pydantic 抽離出來是絕對正確的選擇。

理由如下：

1. 比對邏輯太複雜，且具遞迴性 您在計畫書裡定義的 OpponentPattern 支援 any_of, all_of, not_ 等組合模式。如果把這些邏輯寫在 BaseModel 裡，程式碼會變得很肥大難讀。
2. 依賴外部定義 (Dependencies) 當 pattern.suit == "major" 時，matches 方法需要去查閱系統定義的 SUIT_GROUPS 才能知道這包含紅心和黑桃。如果寫在 BaseModel 裡，您很難乾淨地把 SUIT_GROUPS 傳遞進去給每一個節點。
3. Phase 4 的模擬引擎需求 到了 Phase 4 (AI 與模擬引擎)，您不只要判斷「模式是否匹配」，您可能還要「生成符合模式的叫品」。這時候如果邏輯都綁死在 Pydantic 裡，會造成嚴重的架構瓶頸。

### 具體修改建議
讓模型變「貧血 (純資料)」： 在 models/context.py 裡，純粹定義資料結構，把 def matches() 砍掉。

```python
class OpponentPattern(BaseModel):
    simple: Optional[str] = None
    bid: Optional[str] = None
    # ... (保留所有欄位) ...
    # 刪除 matches 方法
```

在 core 目錄下建立專屬的引擎： 新增一個 core/matcher.py 或 core/evaluator.py：

```python
# bbdsl/core/matcher.py
from bbdsl.models.context import OpponentPattern
class PatternMatcher:
    def __init__(self, definitions: dict):
        """可以把全域的名詞定義注入進來"""
        self.definitions = definitions
    def matches(self, pattern: OpponentPattern, action: dict) -> bool:
        # 實作比對邏輯...
        if pattern.simple:
            return self._match_simple(pattern.simple, action)
        # ...遞迴處理 any_of, all_of
```

這個改動能確保您的 models 目錄永遠輕量、乾淨，只做「讀取 YAML 和驗證格式」這件事，這是構建健壯編譯器的黃金法則。



  6. BML 匯入的範圍定義

  BML 本身沒有正式規格（只有 convention），不同作者的 BML 寫法差異很大。計畫書目標「HCP/花色長度正確提取率 > 80%」——這個
   80% 是基於哪些 BML 樣本測量的？有沒有預定的測試語料庫？

  這是一個切中要害的 QA 測試策略問題！

其實在目前的開源橋牌界，沒有一個官方標準的 BML 測試語料庫（Corpus）。計畫書中提到的「正確提取率 > 80%」是一個「工程啟發式（Heuristic）的目標」，也就是說，這 80% 是需要我們自己建立 Benchmark (基準測試) 來測量的。

為了讓這個 80% 有科學依據，且確保您的 BML 匯入器真的實用，我建議採取以下策略來定義這個範圍和測試語料庫：

1. 建立「黃金標準語料庫 (Golden Corpus)」
在 Phase 1，您不需要去網路上隨便找幾百份格式千奇百怪的 BML 來為難自己。您只需要在 tests/fixtures/bml_samples/ 裡面放入 3~5 份最具代表性的 BML 檔案 即可。

推薦的樣本來源：

* 原版 BML 官方範例：到 BML 的發明者 (biefnes 或 kalkin) 的 GitHub repo 中，直接下載他們提供的 SAYC.bml 和 Precision.bml。這是最正統的寫法。
* 常見的社群變體：去 GitHub 搜尋 .bml 檔案，挑選 1~2 份星數較高、排版比較整齊的 2/1 GF (二蓋一) 制度檔。
* 您自己手寫的測試檔：為了測試極限，寫一份包含各種寫法（如 12-14 HCP、12--14、12+、5+H、5!h）的假檔案。

2. 什麼是「正確提取率 > 80%」？（如何測量）
這個指標的定義不該是「80% 的檔案能匯入」，而是「對於語料庫中的每一條規則，有多大比例的文字描述被成功轉化為結構化屬性」。

具體的測量與測試方法如下：

* 人工標註 (Ground Truth)：您先看著語料庫裡的某 100 條 BML 叫品，人工算一下裡面其實包含了 40 個 HCP 條件和 30 個花色長度條件。
* 程式測試：跑您的 bml_importer.py，看看它用 Regular Expression (正規表達式) 抓出了幾個。
* 達標標準：如果您的程式成功抓出 32 個 HCP (32/40 = 80%) 和 24 個花色長度 (24/30 = 80%)，剩下的它看不懂、被放到 _unresolved (或 unresolved=True) 裡面去，這樣就算是完成了 Phase 1 的目標！

3. 給您的實務架構建議
在實作 BML 匯入器時，千萬不要寫一個巨大無比、想要解析所有自然語言的 NLP 引擎。

請採用 「管線過濾策略 (Pipeline & Heuristics)」：

* 先寫 3 條最基本的 Regex（抓 XX-YY 點力、抓 N+! 花色長度、抓 BAL/GF/INV 等關鍵字）。
* 用這些 Regex 去跑您的語料庫，看能抓出多少。
* 如果只有 50%，那就再加 2 條 Regex 去處理另一種常見寫法。
* 一旦達到 80%，馬上停手！

剩下的 20%（比如作者用很口語的英文寫 "a bad 12 to a good 14 with a singleton club"）是 Regex 處理不了的。這正是為什麼我們前面討論要保留 unresolved 和 bml_original 的原因。剩下的 20% 就交給使用者匯入後去人工手動修復（或留到 Phase 4 讓 AI 去讀）。

這樣可以確保您的 Phase 1 不會陷入「永遠寫不完 BML 解析器」的泥淖中。

    ```python
        condition: str | None = None

    Node = Union[CallNode, UnresolvedNode]
    ```
