# Python E2E Test 技術架構決策文件

## 文件目的

本文件記錄了從 Unit Test 改版為 E2E Test 時，所有關鍵的技術選型決策、架構設計和實作原則。

---

## 技術棧總覽

| 技術面向 | 選擇 | 替代方案（未選用） |
|---------|------|------------------|
| 測試框架 | pytest | unittest, nose2 |
| HTTP Client | FastAPI TestClient | requests, httpx, aiohttp |
| 資料庫 | PostgreSQL (Testcontainers) | MySQL, SQLite |
| ORM | SQLAlchemy | Django ORM, Peewee |
| Schema 管理 | Alembic | SQLAlchemy create_all(), SQL scripts |
| 認證機制 | JwtHelper (簡化 class) | Adapter Pattern, Mock |
| 測試隔離 | Truncate tables | Transaction rollback, Fresh DB |
| 變數傳遞 | Dict fixture (context) | Class-based Context, Global dict |
| 後端運行 | TestClient (in-process) | Docker Container, 手動啟動 |

## 詳細決策說明

### 1. 測試框架：pytest

**選擇理由**：
- 原本的 UT 已經使用 pytest
- Fixture 機制強大，適合 E2E 測試的複雜依賴注入
- 支援 parametrize，方便測試多組資料
- 社群生態完整

**關鍵特性**：
- `@pytest.fixture` 用於依賴注入
- `scope` 參數控制 fixture 生命週期（session, module, function）
- `autouse=True` 用於 Background 自動執行

---

### 2. HTTP Client：FastAPI TestClient

**選擇理由**：
- 直接在測試進程中運行 ASGI app，無需真實的 HTTP server
- 無需 Docker build 或容器啟動，測試速度大幅提升
- 修改 production code 後立即可測試
- API 與 requests.Session 高度相似，測試代碼幾乎無需修改
- 仍然是完整的 E2E 測試（使用真實 PostgreSQL + 完整 FastAPI app）

**實作方式**：
```python
@pytest.fixture(scope="session")
def app(postgres_container):
    """提供 FastAPI app 實例，配置測試資料庫"""
    from app.main import app
    from app.core.deps import get_db
    
    # 建立測試資料庫引擎
    connection_url = postgres_container.get_connection_url()
    test_engine = create_engine(connection_url)
    TestingSessionLocal = sessionmaker(bind=test_engine)
    
    # Override get_db dependency
    def override_get_db():
        db = TestingSessionLocal()
        try:
            yield db
        finally:
            db.close()
    
    app.dependency_overrides[get_db] = override_get_db
    yield app
    app.dependency_overrides.clear()

@pytest.fixture(scope="session")
def api_client(app):
    """提供 FastAPI TestClient"""
    from fastapi.testclient import TestClient
    return TestClient(app)
```

**優勢**：
- **速度快**：無需 Docker build/啟動（從 60+ 秒降至 <2 秒）
- **開發便捷**：修改代碼立即測試
- **真正的 E2E**：仍使用真實 PostgreSQL，只是 FastAPI 在進程內運行
- **API 相容**：TestClient API 與 requests.Session 高度相似

**為什麼不用 Docker Container**：
- 每次修改代碼都需要 rebuild image
- 啟動時間長（60+ 秒）
- 網路設定複雜（host.docker.internal）
- TestClient 已經提供完整的 E2E 測試能力

### 3. 資料庫：PostgreSQL (透過 Testcontainers)

**選擇理由**：
- 與生產環境一致（不使用 SQLite）
- 透過 Testcontainers 自動管理容器生命週期
- 每次測試執行都是乾淨的環境
- 支援完整的 SQL 特性（不像 SQLite 有限制）

**Testcontainers 設定**：
```python
@pytest.fixture(scope="session")
def postgres_container():
    """啟動 PostgreSQL 容器"""
    with PostgresContainer("postgres:15") as postgres:
        yield postgres

@pytest.fixture(scope="session")
def db_engine(postgres_container):
    """
    創建資料庫引擎並使用 Alembic 執行 migrations
    確保測試環境與生產環境使用相同的 schema 管理方式
    """
    connection_url = postgres_container.get_connection_url()
    engine = create_engine(connection_url)
    
    # ✅ 使用 Alembic 執行 migrations（與生產環境一致）
    from alembic.config import Config
    from alembic import command
    
    alembic_cfg = Config("alembic.ini")
    alembic_cfg.set_main_option("sqlalchemy.url", connection_url)
    command.upgrade(alembic_cfg, "head")
    
    # 額外建立測試用的 E2E models（不在 Alembic migrations 中）
    from app.models import Base
    Base.metadata.create_all(engine)
    
    yield engine
    engine.dispose()
```

**為什麼不用 SQLite**：
- SQLite 與 PostgreSQL 的 SQL 方言不同
- 某些 PostgreSQL 特性在 SQLite 不支援
- E2E 測試應該盡可能接近生產環境

**重要**：測試環境使用 Alembic
- 測試環境也執行 `alembic upgrade head`
- 確保測試與生產環境的 schema 完全一致
- 包括索引、約束、資料遷移邏輯
- 避免 `create_all()` 與 Alembic migrations 不同步的問題

### 4. ORM：SQLAlchemy

**選擇理由**：
- Python 最成熟的 ORM
- 支援多種資料庫
- 與 Alembic 整合良好
- 靈活（可以寫 raw SQL，也可以用 ORM）

**使用 Repository Pattern**：
- 不直接在測試中使用 Session
- 透過 Repository 封裝資料庫操作
- 保持測試程式碼簡潔

```python
class LessonProgressRepository:
    def __init__(self, session: Session):
        self.session = session
    
    def save(self, progress: LessonProgress) -> None:
        self.session.merge(progress)  # 使用 merge 支援 upsert
        self.session.commit()
    
    def find_by_user_and_lesson(self, user_id: str, lesson_id: int):
        return self.session.query(LessonProgress).filter_by(
            user_id=user_id, lesson_id=lesson_id
        ).first()
```

### 5. Schema 管理：Alembic

**選擇理由**：
- SQLAlchemy 官方推薦的 migration 工具
- 支援版本控制
- 可以產生和回滾 migration
- 適合團隊協作

**在測試中的使用**：
```python
# 方法 1: 使用 Alembic migration
from alembic.config import Config
from alembic import command

alembic_cfg = Config("alembic.ini")
command.upgrade(alembic_cfg, "head")

# 方法 2: 簡化版（直接用 create_all）
Base.metadata.create_all(engine)
```

**為什麼不用 create_all()**：
- Alembic 可以管理 schema 變更歷史
- 生產環境也會用 Alembic
- 測試環境應該與生產環境一致

---

### 6. 認證機制：JwtHelper (簡化版)

**選擇理由**：
- 不需要完整的 Adapter Pattern（Java 版本太複雜）
- 測試專用，只需要生成 token 的功能
- 簡單直接

**實作方式**：
```python
class JwtHelper:
    """JWT Token 生成器（測試專用）"""
    
    def __init__(self, secret_key: str = "test-secret-key"):
        self.secret_key = secret_key
    
    def generate_token(self, user_id: str) -> str:
        """生成 JWT Token"""
        payload = {
            "sub": user_id,
            "exp": datetime.utcnow() + timedelta(hours=1)
        }
        return jwt.encode(payload, self.secret_key, algorithm="HS256")

@pytest.fixture(scope="session")
def jwt_helper():
    return JwtHelper()
```

**為什麼不用 Adapter Pattern**：
- Python 測試環境不需要那麼複雜的抽象
- 測試專用的簡單實作更易維護
- 避免過度設計

---

### 7. 測試隔離：每個測試前 Truncate Tables

**選擇理由**：
- 確保每個測試都是乾淨的狀態
- 比 transaction rollback 更可靠（某些 ORM 操作可能繞過 transaction）
- 比每次重建資料庫快

**實作方式**：
```python
@pytest.fixture(autouse=True)
def truncate_tables(db_session):
    """每個測試前清空所有 tables"""
    yield  # 測試執行
    
    # 測試後清空
    for table in reversed(metadata.sorted_tables):
        db_session.execute(table.delete())
    db_session.commit()
```

**為什麼不用 Transaction Rollback**：
- 某些操作（如 CREATE INDEX）無法回滾
- Truncate 更明確，不會有隱藏的副作用

**為什麼不用 Fresh DB**：
- 每次重建資料庫太慢
- Testcontainers 在 session scope，重用更快

---

### 8. 變數傳遞：Dict Fixture (context)

**選擇理由**：
- 簡單直觀，就是一個普通的 dict
- 不需要額外的 class 定義
- pytest fixture 自動管理生命週期

**實作方式**：
```python
@pytest.fixture
def context():
    """測試上下文，用於在步驟間傳遞資料"""
    return {}
```

**使用方式**：
```python
# Given: 儲存 ID
context["Alice.id"] = "user-123"

# When: 取得 ID
user_id = context["Alice.id"]

# When: 儲存 response
context["last_response"] = response

# Then: 取得 response
response = context["last_response"]
```

**為什麼不用 Class-based Context**：
- Dict 已經足夠
- 不需要額外的 getter/setter 方法
- Python 的 dict 本身就很強大

---

### 9. Base URL 設定：在 conftest.py 中定義

**選擇理由**：
- 集中管理，方便修改
- 可以從環境變數讀取
- 所有測試共用同一個設定

**實作方式**：
```python
@pytest.fixture
def api_client():
    session = requests.Session()
    # 從環境變數或設定檔讀取
    base_url = os.getenv("API_BASE_URL", "http://localhost:8000")
    session.base_url = base_url
    # ... (包裝 request 方法)
    yield session
```

**為什麼不在 Handler 中寫死**：
- 不同環境可能有不同的 URL
- 測試應該可以在本地和 CI/CD 環境執行

---

### 10. Testcontainers Scope：Session

**選擇理由**：
- PostgreSQL 啟動較慢（約 5-10 秒）
- Session scope 只啟動一次，所有測試共用
- 透過 truncate tables 確保測試隔離

**實作方式**：
```python
@pytest.fixture(scope="session")
def postgres_container():
    """整個測試 session 只啟動一次"""
    with PostgresContainer("postgres:15") as postgres:
        yield postgres
```

**為什麼不用 Function Scope**：
- 太慢，每個測試都要啟動容器
- 不必要，truncate tables 已經足夠隔離

---

### 11. 後端服務運行：FastAPI TestClient (In-Process)

**選擇理由**：
- **極速啟動**：無需 Docker build 或容器啟動（<2 秒 vs 60+ 秒）
- **開發便捷**：修改 production code 立即可測試，無需 rebuild
- **簡化架構**：只需要 PostgreSQL Container，FastAPI 在測試進程中運行
- **仍是真正的 E2E**：使用真實 PostgreSQL + 完整 FastAPI app
- **API 相容**：TestClient API 與 requests 幾乎完全相同

**實作方式**：
```python
@pytest.fixture(scope="session")
def app(postgres_container):
    """提供 FastAPI app 實例，配置測試資料庫"""
    from app.main import app
    from app.core.deps import get_db
    
    # Override database dependency
    connection_url = postgres_container.get_connection_url()
    test_engine = create_engine(connection_url)
    TestingSessionLocal = sessionmaker(bind=test_engine)
    
    def override_get_db():
        db = TestingSessionLocal()
        try:
            yield db
        finally:
            db.close()
    
    app.dependency_overrides[get_db] = override_get_db
    yield app
    app.dependency_overrides.clear()

@pytest.fixture(scope="session")
def api_client(app):
    """提供 FastAPI TestClient"""
    from fastapi.testclient import TestClient
    return TestClient(app)
```

**為什麼不用 Docker Container**：
- Docker 方案需要：
  - 每次修改代碼都要 rebuild image
  - 等待容器啟動（60+ 秒）
  - 複雜的網路設定（host.docker.internal）
  - CI/CD 需要 Docker-in-Docker
- TestClient 已經提供完整的 E2E 測試能力
- 測試速度提升 30 倍以上（從 60+ 秒降至 <2 秒）

### 12. Entity ID 儲存策略：{natural_key}.id

**選擇理由**：
- 與 Java 版本保持一致
- 自然鍵（如 "Alice"）易讀易理解
- 支援後續步驟使用同一個用戶

**實作方式**：
```python
# Given: 創建 Entity 後儲存
progress = LessonProgress(user_id="Alice", ...)
repository.save(progress)
context["Alice.id"] = progress.user_id

# When: 使用儲存的 ID
user_id = context["Alice.id"]
token = jwt_helper.generate_token(user_id)
```

**格式規則**：
- 格式：`{natural_key}.id`
- natural_key 通常來自 Gherkin 中的字串參數
- 如 `"Alice"` → `context["Alice.id"]`

### 13. 專案結構：tests/e2e/

**選擇理由**：
- 與主專案分離但在同一個 repo
- 可以共用部分程式碼（如 models 定義）
- 符合 Python 慣例

**建議結構**：
```
project/
├── src/                    # 主專案程式碼
│   ├── controllers/
│   ├── services/
│   └── models/
├── tests/
│   ├── unit/              # Unit Tests
│   └── e2e/               # E2E Tests
│       ├── conftest.py    # Fixtures 定義
│       ├── models/        # SQLAlchemy Models
│       │   └── lesson_progress.py
│       ├── repositories/  # Repository 實作
│       │   └── lesson_progress_repository.py
│       └── test_*.py      # 測試檔案
├── alembic/               # Migration scripts
├── requirements.txt

```

**為什麼不獨立專案**：
- 需要共用 models 定義
- 部署時不需要分開管理
- 方便 CI/CD 整合

## API Spec 參考策略

### 核心原則

**必須參考 api.yml**：
- 所有 Command 和 Query Handler 都要參考 api.yml
- 欄位命名使用 camelCase（與 api.yml 一致）
- Request/Response 結構完全依照 api.yml

### 參考流程

1. **找到對應的 Endpoint**：
   - 在 api.yml 的 `paths` 下搜尋
   - 使用 Gherkin 的動詞和名詞作為關鍵字

2. **確認 HTTP Method**：
   - Command: POST, PUT, PATCH, DELETE
   - Query: GET

3. **構建 Request Body**（Command）：
   - 參考 `requestBody.content.application/json.schema.properties`
   - 使用 camelCase 欄位名稱

4. **驗證 Response**（Query）：
   - 參考 `responses['200'].content.application/json.schema`
   - 驗證欄位存在和型別

### 範例

**api.yml**：
```yaml
/lesson-progress/update-video-progress:
  post:
    requestBody:
      properties:
        lessonId:
          type: integer
        progress:
          type: integer
```

**生成的測試程式碼**：
```python
response = api_client.post(
    "/api/lesson-progress/update-video-progress",
    json={
        "lessonId": 1,      # camelCase
        "progress": 80
    }
)
```

## 測試執行流程

### 1. 環境啟動（Session Scope）

```
pytest 啟動
  ↓
啟動 PostgreSQL Container (session)
  ↓
創建 DB Engine (session)
  ↓
執行 Alembic Migration
  ↓
啟動後端服務 Container (session)
  ↓
等待服務就緒
```

### 2. 測試執行（Function Scope）

```
測試開始
  ↓
創建 DB Session (function)
  ↓
創建 Context Dict (function)
  ↓
執行測試
  ↓
Truncate Tables
  ↓
關閉 Session
  ↓
測試結束
```

### 3. 環境清理（Session End）

```
所有測試完成
  ↓
停止後端服務 Container
  ↓
停止 PostgreSQL Container
  ↓
清理資源
```

## 紅燈階段的特殊性

### E2E 紅燈 vs UT 紅燈

| 面向 | Unit Test 紅燈 | E2E Test 紅燈 |
|------|---------------|--------------|
| 測試失敗原因 | NotImplementedError | HTTP 404 |
| 需要定義 | Entity, Repository, Service 介面 | Model, Repository, Fixtures |
| 不需要定義 | 業務邏輯 | 後端 API (Controller/Service) |
| 資料庫 | FakeRepository | 真實 PostgreSQL |
| 測試可執行 | 是（但失敗） | 是（但失敗） |

### 紅燈階段要做的事

1. **完整實作測試程式碼**
2. **定義 SQLAlchemy Models**
3. **定義 Repository 類別**
4. **設定 Fixtures（conftest.py）**
5. **啟動測試基礎設施**

### 紅燈階段不做的事

1. **不實作後端 API**（Controller/Service）
2. **不讓測試通過**
3. **不跳過基礎設施定義**

## 常見問題與解決方案

### Q1: 測試執行太慢？

**解決方案**：
- 確保使用 session scope 的容器
- 使用 truncate 而非重建資料庫
- 考慮使用 pytest-xdist 平行執行測試

### Q2: PostgreSQL 啟動失敗？

**解決方案**：
- 確認 Docker 已啟動
- 檢查 port 是否被佔用
- 增加 Testcontainers 的啟動等待時間

### Q3: API response 欄位名稱不一致？

**解決方案**：
- 嚴格遵守 api.yml 定義
- 使用 camelCase（前端慣例）
- 在後端使用序列化器統一處理

### Q4: 測試之間有資料污染？

**解決方案**：
- 確認 truncate_tables fixture 有正確執行
- 檢查是否有 transaction 未提交
- 考慮使用 `autouse=True`

### Q5: JWT token 驗證失敗？

**解決方案**：
- 確認後端和測試使用相同的 secret key
- 檢查 token 是否過期
- 確認 Authorization header 格式正確

## 最佳實踐

### 1. Fixture 設計

- 使用適當的 scope（session/module/function）
- 善用 `autouse=True` 處理通用設定
- Fixture 命名清晰表達用途

### 2. Repository Pattern

- 所有資料庫操作透過 Repository
- 測試中不直接使用 Session
- Repository 方法命名清晰（如 `find_by_user_and_lesson`）

### 3. Context 使用

- 統一使用 `{natural_key}.id` 格式儲存 ID
- 使用 `last_response` 儲存最後的 HTTP response
- 避免在 context 中儲存過多資料

### 4. API 測試

- 嚴格參考 api.yml
- 不在 Command Handler 中驗證 response
- Success/Failure 由專門的 Handler 處理

### 5. 測試組織

- 每個 Gherkin Rule 對應一個 Test Class
- Background 使用適當層級的 fixture
- 測試方法命名與 Gherkin Example 一致

## 總結

本 Python E2E 測試架構的核心特點：

1. **真實環境**：使用真實的 PostgreSQL 和後端服務（容器化）
2. **簡化設計**：避免過度設計，使用 Python 慣用方式
3. **與生產一致**：測試環境盡可能接近生產環境
4. **易於維護**：清晰的結構和命名規則
5. **完整隔離**：每個測試都是乾淨的狀態

這些決策確保了 E2E 測試的可靠性、可維護性和執行效率。
