# E2E Test Implementation - 紅燈生成器

## Role
將 E2E 測試樣板（純註解）轉換為可執行的 E2E 測試程式碼，依照註解中的 Handler Prompt 指引生成對應的程式碼，生成紅燈測試。

## Core Task
E2E 測試樣板（註解）→ 可執行 E2E 測試程式碼（紅燈）

## Input
1. 測試方法（包含註解樣板，來自 01-Feature-to-測試樣板.md）
2. DBML（Aggregate 定義）
3. API Spec（api.yml）
4. Tech Stack（Python + pytest + requests + SQLAlchemy + Testcontainers + PostgreSQL）
5. Handler Prompts（handlers/ 資料夾中的所有 Handler）

## Output

### 1. 測試程式碼
完整的 E2E 測試程式碼，包含：
- 必要的 import
- pytest fixtures 注入
- 完整的測試方法實作
- 所有測試邏輯完整實作

### 2. 基礎設施定義
**如果測試中需要用到的基礎設施尚不存在**，則按以下規則定義：

#### 生產環境基礎設施 (在 `app/` 中定義)
- **SQLAlchemy Models**：放在 `app/models/`
- **Repositories**：放在 `app/repositories/`

#### 測試專用基礎設施 (在 `tests/e2e/` 中定義)
- **Fixtures**：在 `conftest.py` 中定義 (db_session, api_client, context, jwt_helper)
- **測試文件組織**：
  - **正式功能測試** → `tests/e2e/test_*.py`（如 `test_update_video_progress.py`）
    - ✅ **必須放在 `tests/e2e/` 正下方**
    - ❌ **絕對不要放在 `tests/e2e/walking_skeleton/`**
    - 這些是測試真實業務功能的測試案例
  - **基礎建設驗證** → `tests/e2e/walking_skeleton/test_*.py`（如 `test_walking_skeleton.py`）
    - 只用於驗證測試基礎設施是否正常運作
    - 不屬於正式業務測試

**為什麼要放在 app/?**
- 這些是生產環境也需要的程式碼
- 綠燈階段實作 API 時，Controller/Service 會直接使用這些 Models 和 Repositories
- 測試只是使用它們，不應該擁有它們

**關鍵原則：僅定義介面和基礎設施，不實作後端業務邏輯**
- Model 和 Repository 類別定義完整（放在 `app/`）
- 後端 API (Controller/Service) 不實作
- E2E 測試會因為 HTTP 404 而失敗（後端 API 尚未實作）

## 重要規範

### API JSON 欄位命名規則

- ✅ **所有 API Request/Response 的 JSON 欄位必須使用 snake_case**
- ✅ Gherkin Feature File 使用自然語言，不受此限制
- ❌ 絕對不要在 API JSON 中使用 camelCase

範例：
```python
# ✅ 正確
json={"lesson_id": 1, "progress": 80}
assert data["user_id"] == "Alice"

# ❌ 錯誤
json={"lessonId": 1, "progress": 80}
assert data["userId"] == "Alice"
```

## 紅燈階段的核心原則

### E2E 測試的紅燈特色

與 Unit Test 的關鍵差異：

| 面向 | Unit Test 紅燈 | E2E Test 紅燈 |
|------|---------------|--------------|
| 測試失敗原因 | Service 方法拋出 NotImplementedError | HTTP 404 Not Found |
| 需要定義的東西 | Aggregate, Repository, Service 介面 | SQLAlchemy Model, Repository, Fixtures |
| 不需要定義的東西 | 業務邏輯 | 後端 API (Controller/Service) |
| 資料庫 | FakeRepository（不需要真實 DB） | 真實 PostgreSQL（需要 Testcontainers） |

### ✅ 要做的事
1. **完整實作測試程式碼**：測試邏輯必須完整且正確，放在 `tests/e2e/test_*.py`（**必須在 `tests/e2e/` 正下方**）
2. **定義 SQLAlchemy Models**：在 `app/models/` 中創建 Aggregate 的 ORM 定義
3. **定義 Repository 類別**：在 `app/repositories/` 中創建 Repository
4. **設定測試基礎設施**：在 `tests/e2e/conftest.py` 中定義必要的 fixtures
5. **啟動測試資料庫**：使用 Testcontainers 啟動 PostgreSQL

### ❌ 不要做的事
1. **不要實作後端 API**：Controller 和 Service 不實作
2. **不要讓測試通過**：測試應該因為 HTTP 404 而失敗（紅燈）
3. **不要跳過基礎設施定義**：測試需要的 Model、Repository、Fixtures 必須定義
4. **不要將正式功能測試放在 `walking_skeleton/`**：正式功能測試必須放在 `tests/e2e/test_*.py`，只有基礎建設驗證測試才放在 `tests/e2e/walking_skeleton/test_*.py`

### 為什麼要這樣？
這是 TDD 的核心流程：
1. **紅燈**：寫測試 + 定義基礎設施（測試失敗：HTTP 404）← 我們現在在這
2. **綠燈**：實作後端 API（測試通過）
3. **重構**：優化程式碼品質（測試持續通過）

### Step 1: 讀取測試樣板

### Step 2: 逐步生成程式碼
根據每個註解區塊的 Handler Prompt 生成對應的程式碼

#### 範例流程

**Given 區塊**：
- Handler: Aggregate-Given-Handler.md
- 生成: 使用 SQLAlchemy Model 創建 entity → repository.save() → 儲存 ID 到 context

**When 區塊**：
- Handler: Command-Handler.md
- 生成: 從 context 取得 user_id → jwt_helper 生成 token → 構建 request body → api_client.post() → 儲存 response 到 context

**Then 區塊**：
- Handler: Success-Failure-Handler.md
- 生成: 從 context 取得 response → assert response.status_code in [200, 201, 204]

**And 區塊**：
- Handler: Aggregate-Then-Handler.md
- 生成: 使用 repository 查詢 entity → assert entity.field == expected

### Step 3: 生成測試基礎設施
包含必要的 fixtures、models、repositories

### Output 1（測試程式碼 - tests/e2e/test_update_video_progress.py）
**重要**：測試檔案必須放在 `tests/e2e/` 正下方，不要放在 `walking_skeleton/`

```python
import pytest
from app.repositories.lesson_progress_repository import LessonProgressRepository
from app.models.lesson_progress import LessonProgress


class Test影片進度必須單調遞增:
    """
    Rule: 影片進度必須單調遞增
    """
    
    def test_成功增加影片進度(self, db_session, api_client, context, jwt_helper):
        # Arrange: 初始化 repository
        repository = LessonProgressRepository(db_session)
        
        # Given 用戶 "Alice" 在課程 1 的進度為 70%，狀態為 "進行中"
        # [使用 Aggregate-Given-Handler.md]
        progress = LessonProgress(
            user_id="Alice",
            lesson_id=1,
            progress=70,
            status="IN_PROGRESS"
        )
        repository.save(progress)
        context["Alice.id"] = progress.user_id
        
        # When 用戶 "Alice" 更新課程 1 的影片進度為 80%
        # [使用 Command-Handler.md]
        user_id = context["Alice.id"]
        token = jwt_helper.generate_token(user_id)
        
        response = api_client.post(
            "/api/lesson-progress/update-video-progress",
            headers={"Authorization": f"Bearer {token}"},
            json={"lessonId": 1, "progress": 80}
        )
        context["last_response"] = response
        
        # Then 操作成功
        # [使用 Success-Failure-Handler.md]
        assert response.status_code in [200, 201, 204]
        
        # And 用戶 "Alice" 在課程 1 的進度應為 80%
        # [使用 Aggregate-Then-Handler.md]
        updated_progress = repository.find_by_user_and_lesson(user_id="Alice", lesson_id=1)
        assert updated_progress.progress == 80
```

### Output 3（Repository - app/repositories/lesson_progress_repository.py）

```python
from typing import Optional
from sqlalchemy.orm import Session
from app.models.lesson_progress import LessonProgress


class LessonProgressRepository:
    """課程進度 Repository - 使用 SQLAlchemy"""
    
    def __init__(self, session: Session):
        self.session = session
    
    def save(self, progress: LessonProgress) -> None:
        """保存課程進度到資料庫"""
        self.session.merge(progress)  # 使用 merge 處理 upsert
        self.session.commit()
    
    def find_by_user_and_lesson(self, user_id: str, lesson_id: int) -> Optional[LessonProgress]:
        """查詢課程進度"""
        return self.session.query(LessonProgress).filter_by(
            user_id=user_id,
            lesson_id=lesson_id
        ).first()
```

class JwtHelper:
    """JWT Token 生成器（測試專用）"""

## Critical Rules

### R1: 測試程式碼必須完整
測試邏輯必須完整實作，不能有 `pass` 或空白。

```python
# ✅ 正確：完整的測試邏輯
def test_example(self, db_session, api_client, context, jwt_helper):
    repository = ExampleRepository(db_session)
    entity = Example(id=1, name="test")
    repository.save(entity)
    context["test.id"] = entity.id
    
    token = jwt_helper.generate_token(entity.id)
    response = api_client.post("/api/example", 
                               headers={"Authorization": f"Bearer {token}"},
                               json={"data": "value"})
    context["last_response"] = response
    
    assert response.status_code == 200
    result = repository.find(id=1)
    assert result.name == "expected"

# ❌ 錯誤：測試邏輯不完整
def test_example(self, db_session, api_client, context, jwt_helper):
    pass
```

## 與 Unit Test 紅燈的比較

| 面向 | Unit Test 紅燈 | E2E Test 紅燈 |
|------|---------------|--------------|
| 測試對象 | Service 方法 | HTTP API |
| 資料庫 | FakeRepository（記憶體） | PostgreSQL (Testcontainers) |
| 失敗原因 | NotImplementedError | HTTP 404 |
| 需要定義 | Entity, Repository, Service 介面 | SQLAlchemy Model, Repository, Fixtures |
| Models 位置 | `app/models/` | `app/models/` |
| Repositories 位置 | `app/repositories/` | `app/repositories/` |
| 測試文件位置 | `tests/unit/` | `tests/e2e/` |
| 不需要定義 | 業務邏輯 | 後端 API (Controller/Service) |
| 測試執行環境 | 本地記憶體 | 真實容器化環境 |

**關鍵共同點**：
- ✅ Models 和 Repositories **都放在 `app/` 中**（生產環境需要）
- ✅ 測試只是使用這些組件，不擁有它們
